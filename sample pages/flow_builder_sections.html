Stage Library

Every process needs the right stage type

Initial stages capture requests. Standard stages route work through teams. Decision stages evaluate conditions. Parallel stages split execution. Final stages close and archive records. Each stage carries its own form, assignment logic, SLA, and action set.

Example: An HR onboarding flow uses 1 initial stage, 6 standard stages (approvals, IT provisioning, payroll, training), and 1 final stage that closes and archives the record.
1Initial stage — capture submissionEntry
2Standard stage — validate, assign, actProcess
3Decision stage — evaluate conditionsConditional
4Parallel stage — concurrent branchesParallel
5Final stage — close, notify, archiveComplete
Conditional Branching

Every transition can carry execution logic

Each connection between stages carries optional conditions. Use AND/OR groups to evaluate form fields, document state, ownership, time elapsed, org attributes, or external API results. The engine decides the path at runtime — not at design time.

Example: After manager review — IF approved → finance stage. IF rejected → return with comments. IF no response in 48 hours → auto-escalate to department head.
Form field value — Equals, contains, greater than, less than, empty, regex
Document status — In stage, approved, rejected, overdue
Owner identity — Assigned role, user, department, previous executor
Time elapsed — Since start, since stage entry, exact date, working hours only
Org structure — Requester department, manager, org level
External data — API response value, webhook payload, lookup result
Parallel Tasks

Run multiple branches simultaneously

Split one process into parallel branches and define how they converge. Choose from all-branches, first-complete, or N-of-M join logic to eliminate artificial sequencing and cut cycle time.

Example: Leave approved → payroll update and employee notification run in parallel. Neither branch waits for the other. Both complete independently.
All branches complete — Process advances only when every parallel branch has finished
First branch completes — Advance on the fastest outcome; remaining branches are cancelled
N-of-M branches complete — Define a threshold — e.g. 3 of 5 approvers must act
Timed join with fallback — If not all branches complete within a deadline, advance with a fallback action
Start Triggers

A process can start from six different event types

Launch workflows manually, on a schedule, from another process, from an inbound communication, from an API event, or from a field value change. This makes the orchestrator suitable for both human-centric and event-driven processes.

Example: Every Monday at 8am, a weekly team checklist process starts automatically, assigns tasks to each lead, and escalates uncompleted items by Wednesday noon.
Manual form submission — User fills and submits a form to start the process
Scheduled trigger — Cron-style schedule: daily, weekly, monthly
Process-to-process — Child process starts when a parent stage completes
Inbound communication — Email, SMS, IVR call, or WhatsApp message received
External API event — Webhook payload from any external system fires the trigger
Field value change — A record update in another form or process triggers a new instance
Organization & Access

Keep every process organized, versioned, and access-controlled

Group workflows into folders by department. Control who can start, view, edit, or administer each process. Publish new versions without interrupting live instances — changes apply to new instances only.

Example: Finance processes are in a locked folder. Only the Finance process owner can publish changes. Live expense approval cases continue on the previous version uninterrupted.
Folder-based organization — Group by department, use case, or business unit
Process owner role — Designated owner with monitoring and publish rights
Version control — New versions deploy to new instances only — live cases unaffected
Role-based permissions — Separate start, view, edit, and admin rights per process
Graph and list views — Visual canvas or tabular view per user preference
Process health dashboard — Live status of all deployed processes across the org

Events, Actions & Automation

does automatically.

Every significant runtime event can trigger one or more actions — stage entry, stage exit, condition match, timeout, inbound communication, or escalation threshold. No custom code. No external scheduler.

Notification Actions

Notify the right people at the right moment

Send email, SMS, fax, or internal notifications when a stage is entered, approved, rejected, completed, or overdue. Use template variables to personalize messages with live form data and routing context.

Example: Leave request rejected → the requester receives an SMS with the rejection reason and a link to resubmit within 30 seconds. No manual follow-up.
📣

Notification channels available

Email — HTML template with form field variables
SMS — Instant delivery via native SMS module
Internal notification — In-platform alert to any user or role
Fax — Direct fax via native fax server integration
Trigger: Stage entry / exit / approval / rejection / overdue
Process Chaining

One workflow completion can start another automatically

When a stage completes or a final outcome is reached, trigger a downstream process and pass form data between parent and child. Build multi-domain flows without monolithic processes.

Example: Sales process closes → onboarding starts automatically → on completion, billing setup starts. Three processes. Zero manual handoffs.
🔗

Process chain execution

Trigger point — Stage completion or final stage reached
Data passing — Form fields mapped to child process input
Child process starts — Independently tracked instance created
Parent status — Parent case remains linked for audit visibility
Depth — Multi-level chains supported (A → B → C → ...)
Scheduled Automation

Processes that start themselves — on a schedule

Define recurring processes for weekly reviews, monthly compliance checks, quarterly cycles, or routine reminders. The schedule is managed inside the platform — no cron jobs, no external scripts, no developer.

Example: Every Monday at 8am, a weekly team checklist starts automatically. Tasks are assigned to each team lead. Uncompleted items escalate to the manager by Wednesday noon.
⏱

Schedule configuration options

Daily — Fixed time, every day or on specific weekdays
Weekly — Day of week + time, repeating
Monthly — Day of month + time, with end-of-month handling
Custom interval — Every N hours or N days
Calendar-aware — Skip weekends or public holidays
External System Actions

Push and pull data across any connected system

Fire REST calls or webhooks on any stage event. Send filtered payloads to external endpoints and write API responses back into workflow fields. True bi-directional orchestration — not one-way export.

Example: IT access request approved → API call provisions the account in Active Directory → response confirms provisioning → form field "Account Status" auto-updates to Active.
🔌

API action capabilities

REST POST/GET/PUT/PATCH — Any HTTPS endpoint
Payload filtering — Send a subset of form fields, not the full document
Response mapping — Write API response values back into form fields
Auth support — Bearer token, API key, Basic auth
Retry policy — Configurable retry on failure with alert on final failure
SLA & Escalation Engine

Missed deadlines escalate automatically

Define SLA timers on any stage and configure multi-level escalation chains. The engine reassigns tasks, notifies managers, increases priority, or logs compliance events when deadlines are missed — without any manual intervention.

Example: IT ticket not actioned in 4h → reminder sent. 8h → escalate to team lead. 24h → escalate to IT manager and flag for SLA breach report.
⚡

Escalation chain levels

Level 0 — Reminder — Notify current assignee before deadline
Level 1 — Reassign — Move task to backup assignee or team lead
Level 2 — Escalate — Notify manager + change task priority
Level 3 — Compliance event — Log SLA breach for audit + trigger corrective workflow
Calendar-aware SLA — Count only working hours, not calendar time
